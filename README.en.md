# OpenStack SIG Development Tool

`oos` (openEuler OpenStack SIG) is a command-line tool provided by the OpenStack SIG. It offers several capabilities for OpenStack SIG development, including:

1. Automatically generate and update RPM Spec files
2. Automatically analyze OpenStack package dependencies
3. Automatically submit Spec PRs to the openEuler community
4. Obtain a list of OpenStack SIG CI-failed PRs
5. Create branches for software repositories

`oos` is continuously evolving. Users can install the stable version published on PyPI:

```shell script
pip install openstack-sig-tool
```

## Automatically Generate, Update, and Build RPM Spec Files

The tool supports generating a single RPM Spec file or batch generating RPM Spec files, with optional push and PR submission to the openEuler community.

- Generate an RPM Spec file for a package

```shell script
oos spec create --name stevedore --version 1.28.0
```

- Update an RPM Spec file for a package

```shell script
oos spec update --name stevedore --version 2.0.0
```

Other supported options are:

```shell script
-n, --name
    Specify the package name
-v, --version
    Specify the package version
-s, --special
    When this parameter is used, search for the Spec file in the current directory and replace Version/Release/changelog in the old Spec file
-o, --output
    Specify the output Spec file name
-d, --download
    When this parameter is specified, download the source URL file in the current directory
-r, --replace
    When this parameter is specified, replace the Source URL instead of appending to it
```

- Build an RPM package

```shell script
oos spec build stevedore
```

- Copy files from a software repository to the rpmbuild directory

```shell script
oos spec cp
```

Other supported options are:

```shell script
-c, --clear
    Delete the rpmbuild directory
-b, --build
    Execute `rpmbuild -ba pkg.spec` after copying the files
-i, --install-requires
    Automatically install missing dependency packages listed in the Spec file
```

## Automatically Analyze OpenStack Package Dependencies

Using OpenStack Train as an example:

1. Run the script to generate a cache file. By default it is stored in the `train_cached_file` directory.

    ```shell script
    cd tools/oos/scripts
    export OOS_CONF_DIR=$(pwd)/etc
    python3 generate_dependence.py train
    ```

    By default, this command generates all OpenStack services supported by the Train version SIG. Users can also specify the required OpenStack projects, for example:

    ```shell script
    python3 generate_dependence.py --projects nova,cinder train
    ```

2. Use the `oos` command to generate the dependency analysis result.

    ```shell script
    oos dependence generate train_cached_file
    ```

    Other supported options are:

    ```shell script
    -c, --compare
        Whether to compare the result with the openEuler community repository and generate recommendations
    -cb, --compare-branch
        Specify the openEuler repository branch to compare with, default is master
    -cf, --compare-from
        Specify the base branch for openEuler comparison to suggest sync branch recommendations, default is master
    -t, --token
        If `-c` is used, specify the AtomGit token at the same time; otherwise AtomGit may reject access.
        Alternatively, set the `ATOMGIT_PAT` environment variable.
    -o, --output
        Specify the output file name generated by the command, default is `result.csv`
    ```

    After the command runs, one result file will be generated in the root directory by default: `result.csv`.

3. Use the `oos` command to compare whether versions in different openEuler software repository branches meet the dependency requirements.

    If a repository reports an error due to network issues, the repository information is recorded in the last two lines.
    The first line lists the dependency file name, which can be reprocessed with the `-p` option.
    The second line lists the openEuler repository name and a simple reason for the error, which is convenient for checking repository status.

    ```shell script
    oos dependence compare train_cached_file
    ```

    Other supported options are:

    ```shell script
    -b, --branches
        Specify the openEuler repository branches to compare, default is master; multiple branches must be separated by spaces
    -o, --output
        Specify the output file name, default is `compare_result.csv`
    -r, --release
        Specify the release name, such as wallaby or antelope, to obtain package versions from the OpenStack release webpage
    -p, --packages
        Specify the package names to analyze without suffixes; `.json` will be added automatically
    -a, --append
        Append mode; by default, the file is overwritten
    -t, --token
        Access token for AtomGit; anonymous access is limited in API calls
    ```

    Example:

    ```shell script
    oos dependence compare 2023.1_cached_file -b 'master openEuler-23.03' -o my_result -r antelope -p 'aodh aodhclient autobahn'
    ```

    Example output and explanation:

    |Name|RepoName|SIG|eq Version|ge Version|lt Version|ne Version|Upper Version|of community|master|status|openEuler-23.03|status|
    |---|---|---|---|---|---|---|---|---|---|---|---|---|
    |asgiref|python-asgiref|sig-python-modules||3.3.2|4|[]|3.5.2|[]|3.7.2|ge-match lt-match downgrade max3.5.2|3.5.2|ge-match lt-match up-match|
    |openstack-cyborg|openstack-cyborg|sig-openstack|10.0.0|||[]||['10.0.0']|[no tar]||8.0.0|specify ['10.0.0']|

    - Name: the JSON file name
    - RepoName: the corresponding openEuler repository name for the package
    - eq/ge/lt/ne/Upper Version: the version constraints in the JSON file's `version_dict`
    - of community: the package version listed on the OpenStack release webpage; `[]` indicates the package is not listed
    - branch and status: the specified branch names such as `master` and `openEuler-23.03`, and the status indicates the comparison result between the package version in that branch and the `eq/ge/lt/ne/Upper Version` requirement

    In the example above:
    `asgiref` requires a version greater than or equal to 3.3.2, less than 4, and an upper bound of 3.5.2.
    `master` has version 3.7.2, which exceeds the upper limit requirement; `openEuler-23.03` has version 3.5.2, which satisfies the requirement.
    `openstack-cyborg` is a community-specified package requiring version 10.0.0. Neither specified branch satisfies the requirement.

4. Use the `oos` command to read file contents line by line, generate a list, and compare specific differences between two lists.
   By default, it generates a `comp_result.csv` file containing the comparison result.

    ```shell script
    oos dependence compare-list file1 file2
    ```

    Other supported options are:

    ```shell script
    -o, --output
        Specify the output file name, default is `comp_result.csv`
    ```

    For example, if `file1` contains `ansible-lint`, `babel`, etc., and `file2` contains `ansible-lint`, `avro-python3`, etc., the comparison result is as follows:

    |Right|Left|
    |---|---|
    |ansible-lint|ansible-lint|
    |***|avro-python3|
    |babel|***|
    |***|crudini|
    |***|dibbler|
    |diskimage-builder|diskimage-builder|
    |future|***|
    |gnocchi|gnocchi|
    |***|google-auth-httplib2|

## Get the OpenStack SIG PR List

This tool can scan PRs in OpenStack SIG-related projects and summarize them into a list.

1. Use the `oos` command to generate a list of PR information.

    ```shell script
    oos repo pr-fetch -r REPO -t atomgit-pat
    ```

    Supported parameters are as follows:

    ```shell script
    -t, --atomgit-pat
        [Optional] Personal AtomGit access token. You can also set `ATOMGIT_PAT` in the environment.
    -r, --repo
        [Optional] Organization repository name. Defaults to all repositories in the organization in the format `openeuler/xxx` or `src-openeuler/xxx`.
    -o, --output
        [Optional] Output file name, default is `prs.yaml`
    ```

    After running the command, a result file named `prs.yaml` will be generated in the current directory by default.

## Create a Software Repository

You can use `oos repo create` to create an openEuler or src-openeuler software repository.

- To create a repository in `openeuler` or `src-openeuler`, you need to provide a `.csv` file containing the repository list or specify a single repository name, along with AtomGit account information:

```shell script
oos repo create --repo autopage:python-autopage -t ATOMGIT_PAT
```

Or specify a `.csv` file to create multiple repositories at once. The file should contain `pypi_name` and `repo_name` columns:

```shell script
oos repo create --repos-file ~/repos.cvs -t ATOMGIT_PAT
```

Supported parameters are as follows:

```shell script
-t, --atomgit-pat
    [Required] Personal AtomGit access token. You can also set `ATOMGIT_PAT` in the environment.
-e, --atomgit-email
    [Optional] Personal AtomGit email address. You can use `ATOMGIT_EMAIL` or, if the account is public, it can be automatically retrieved from the token.
-o, --atomgit-org
    [Optional] AtomGit organization name to which the repository belongs, default is `src-openeuler`
-r, --repo
    [Optional] Software repository name, mutually exclusive with `--repos-file`, in the format `pypi_name:repo_name`
-rf, --repos-file
    [Optional] `.csv` file containing the names of openEuler community software repositories; currently it should contain the `pypi_name` and `repo_name` columns, mutually exclusive with `--repo`
--community-path
    [Optional] Local path to the `openeuler/community` project
-w, --work-branch
    [Optional] Local work branch, default is `openstack-create-repo`
-dp, --do-push
    [Optional] Specify whether to push to the AtomGit repository and submit a PR; if not specified, changes are only committed locally
```

## Create a Branch for a Software Repository

You can use `oos repo branch-create` to create a branch for an openEuler software repository.

- To create a branch for a repository, provide a `.csv` file containing the repositories for which branches need to be created, or specify a single repository name, along with the corresponding branch information and AtomGit account details.
  For example, create the `openEuler-21.09` branch for `openstack-nova`:

```shell script
oos repo branch-create --repo openstack-nova -b openEuler-21.09 protected master -t ATOMGIT_PAT
```

- To create multiple branches in batch, provide a `.csv` file containing repositories or specify a single repository name, along with the new branch information and AtomGit account details.
  For example, create the `openEuler-21.09` branch and the `openEuler-22.03-LTS` multi-branch for repositories listed in `repos.csv`, and submit a PR:

```shell script
oos repo branch-create --repos-file repos.csv -b openEuler-21.09 protected master -b openEuler-22.03-LTS protected openEuler-22.03-LTS-Next -t ATOMGIT_PAT --do-push
```

Supported parameters are as follows:

```shell script
-t, --atomgit-pat
    [Required] Personal AtomGit access token. You can also set `ATOMGIT_PAT` in the environment.
-e, --atomgit-email
    [Optional] Personal AtomGit email address. You can use `ATOMGIT_EMAIL` or, if the account is public, it can be automatically retrieved from the token.
-o, --atomgit-org
    [Optional] AtomGit organization name, default is `src-openeuler`
-r, --repo
    [Optional] Software repository name, mutually exclusive with `--repos-file`
-rf, --repos-file
    [Optional] `.csv` file containing the names of openEuler community software repositories; currently only the `repo_name` column is required, mutually exclusive with `--repo`
-b, --branches
    [Required] Branch information to create for the package. Each branch entry contains the target branch name, branch type (typically `protected`), and the parent branch name. Multiple `-b` options can be used to create multiple branches.
--community-path
    [Optional] Local path to the `openeuler/community` project
-w, --work-branch
    [Optional] Local work branch, default is `openstack-create-branch`
-dp, --do-push
    [Optional] Specify whether to push to the AtomGit repository and submit a PR; if not specified, only the local repository is updated
```

## Submit a Specific PR

You can use `oos repo create-pr -t <ATOMGIT_PAT> -m <comment> -a` to submit the current branch changes from the local repository and create a PR. This feature has some limitations, so please understand the parameters carefully before use.

- This command is mainly intended for RPM build workflows; other scenarios may not be fully adapted.
- The PR title is taken from the latest changelog in the repository's SPEC file, using the second line of the changelog. If no content is provided, the PR body remains empty.
- By default, the command reads `git config --list` to determine the username used when creating the PR and does not support modification.

Supported parameters are as follows:

```shell script
-t, --atomgit-pat
    [Required] Personal AtomGit access token. You can also set `ATOMGIT_PAT` in the environment.
-o, --atomgit-org
    [Optional] AtomGit organization name, default is `src-openeuler`
-b, --remote-branch
    [Optional] Name of the target branch for the PR; by default, the current local branch name is used
-a, --add-commit
    [Optional] Executes `git add .`, `git commit -m <changelog>`, and `git push` to submit the current changes. If this parameter is not provided, no changes are committed, and it is assumed that the changes have already been submitted.
-m, --comment
    [Optional] Add a comment to the PR after submission, such as `/sync branch1`
```

## Merge a Specific PR

You can use `oos repo pr-merge -t <ATOMGIT_PAT> -a <author> -n <num>` to batch merge PRs, which is suitable for LTS version development and for synchronizing PRs to other branches using the `/sync` command.
The merge method adds the comments `/lgtm` and `/approve`.

It is recommended to check the related PRs after running the command, and the related links will be printed in the output.

Supported parameters are as follows:

```shell script
-t, --atomgit-pat
    [Required] Personal AtomGit access token. You can also set `ATOMGIT_PAT` in the environment.
-o, --atpmgit-org
    [Optional] AtomGit organization name, default is `src-openeuler`
-s, --sig
    [Optional] SIG to which the PR belongs, default is `sig-openstack`
-a, --author
    [Optional] PR author, default is `openeuler-sync-bot`
-c, --comment
    [Optional] Add another comment to the PR. Only a single line comment is supported; escape characters such as `\n` are not recognized.
-n, --number
    [Optional] Number of PRs to merge; if not specified, all are merged.
```

## Create Branches in Batch for Version Migration

For the development of a new version that inherits from the previous version, you can use `oos repo community-create-pr -i -r <reference-branch> -a <aim-branch> -s <file-path>` to automatically modify the package YAML files in `community`. This command is only applicable to version inheritance. It should be run in the `sig` directory of the `community` repository. It checks whether each package under `src-openeuler` in a SIG has a certain branch; if so, it creates a branch from `Next`.

According to the branch development principle, the new branch is created from the `Next` branch.

Supported parameters are as follows:

```shell script
-i, --inherit
    [Optional] Mark whether this is an inherited branch creation operation.
-r, --reference
    [Optional] Reference branch. For example, to migrate SP3 from SP2, enter the name of the SP2 branch. This corresponds to the branch created in the YAML file for SP2, and SP3 will be created accordingly.
-a, --aim-branch
    [Optional] Name of the branch to be created.
'-s', '--save-path'
    [Optional] Save the list of repositories created for the current branch to this path. The first line is the branch name, and the remaining lines are repository names.
```

## Initialize an EBS Project for a Multi-Version Branch

Initialize an EBS project.
For the development of a new version that inherits from the previous version, you can use `oos repo ebs-init -p <path>` to automatically write the initialization content for the EBS project. This command is only applicable to version inheritance.
It should be run in the `multi_version` directory of the `release-management` repository. Based on the input file, it adds the initialization content for the EBS project.

The file format is as follows: the first line is the branch name, and the remaining lines are repository names. For example:

```text
Multi-Version_OpenStack-Wallaby_openEuler-22.03-LTS-SP4
PyYAML
ansible-lint
...
```

Supported parameters are as follows:

```shell script
-p, --path
    [Required] Path to the repository list file, where the first line is the branch name and the remaining lines are repository names.
```

## Environment and Dependencies

The `oos spec build` command depends on `rpmbuild`, so you need to install the following packages:

```shell script
yum install rpm-build rpmdevtools
```

In addition, you need to prepare the working directories required by `rpmbuild`. Run the following command:

```shell script
rpmdev-setuptree
```

When executing `oos spec build`, you can specify the `--build-root` parameter pointing to the root directory of the `rpmbuild` working tree. By default, it uses the `rpmbuild/` directory under the current user's home directory.

In addition, to make it easier to use the tool, you can use Docker to quickly build a packaging environment. For details, see the [README](https://atomgit.com/openeuler/openstack/blob/master/tools/docker/README.md) in the `docker/` directory.
